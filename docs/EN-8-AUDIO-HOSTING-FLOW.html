What each node means

1Clip neededbuilt

Happening: Something wants a Portuguese clip hosted — the batch job warming common phrases, or (later) a live user playing one for the first time.

Decided: Batch first; runtime write-back stays OFF for now. Only curated/known phrases are ever hosted — never free-chat or improvised lines.

2Versioned filenameproposed

Happening: Work out the file's name from the phrase, voice, provider — plus a version stamp.

Decided: Add a version/fingerprint so a re-recorded clip gets a NEW name. (Today the name is phrase-only — this is the change that makes “new version” actually ship.)

3Already hosted?built

Happening: Before doing any work, ask the durable home (Verpex) “do you already have this file?” — checking the real store, not the throwaway bucket.

Decided: Idempotency is anchored on Verpex, with a content-type guard so a fake “page found” 200 can’t fool it.

4Skipbuilt

Happening: It’s already hosted — do nothing.

Decided: No re-generate, no re-upload, no re-pull. This is the step that stops the endless loop you asked about.

5Synthesize PCMbuilt

Happening: Not hosted yet → call the AI voice service to make the audio, backing off politely if it’s rate-limited.

Decided: On 429/503 the batch stops cleanly and resumes next run rather than hammering — the exact failure EN-8 exists to reduce.

6Upload to bucketbuilt

Happening: Drop the finished clip into the Supabase bucket.

Decided: The bucket is only a temporary hand-off — NOT the system of record. It gets emptied after pickup.

7Notify Verpex?proposed

Happening: Decide whether to tell Verpex “come get it now” or let it check on its schedule.

Decided: Polling every 15 min is the guaranteed path; an optional “pull now” ping just trims latency — nice-to-have, not required.

8Prod pulls (sole puller)built

Happening: Production’s scheduled job pulls new clips out of the bucket.

Decided: ONLY production pulls. Staging never pulls — this is what removes the “two environments fighting over one bucket” problem.

9Copy to prod /audiobuilt

Happening: The clip lands in production’s real /audio folder — its permanent home.

Decided: Verpex /audio is the durable home; the website deploy already leaves it untouched.

10Confirm copied keysbuilt

Happening: After a successful copy, prod tells Supabase exactly which clips it got.

Decided: “Confirm, then clean up” — deletion is driven by confirmed copies, not a blind timer.

11Delete from bucketbuilt

Happening: Supabase deletes just those confirmed clips, keeping the bucket small.

Decided: Safe now because there is only ONE puller — nothing else still needs them.

12symlink: staging /audio → prod /audioproposed

Happening: Staging’s /audio is a symlink pointing at prod’s real /audio folder (same server). Whatever prod pulls, staging serves the same files instantly — no copy step.

Decided: Symlink over an rsync sync — it’s the more reliable steady state (no ongoing sync process to silently fail, zero drift, always consistent, same-origin). Trade-off accepted: staging can’t hold audio that prod doesn’t.

Before go-live (2 one-time checks): ① Test symlink-following — ln -s <prod>/audio <staging>/audio-test then curl -I …/audio-test/<known>.pcm; a 200 = the host serves through it. ② Protect it from the deploy — change the web-deploy exclude --exclude 'audio/' → --exclude 'audio' (no slash) so a staging deploy’s --delete can’t remove the link; verify a deploy leaves it intact. Staging’s own pull cron stays OFF (prod is sole puller).

13Orphan backstopbuilt

Happening: Once a day, delete anything left in the bucket older than 7 days.

Decided: Pure safety net for clips never confirmed (e.g. a puller was down) so the bucket can’t grow forever.

14Client playback lookupbuilt

Happening: When the app must speak, it checks sources in order — device cache → pinned → Verpex → bucket → live provider — first hit wins, misses fall through.

Decided: Cheapest/fastest first, paid provider last. The provider always works, so nothing breaks even if hosting is incomplete.
