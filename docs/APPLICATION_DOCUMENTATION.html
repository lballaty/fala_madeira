FalaMadeira - Application Architecture & Data Flow

This document provides a comprehensive, end-to-end overview of the FalaMadeira application. Click on the sections below to drill down into specific logical flows, component placements, and data structures.

1. High-Level Architecture

FalaMadeira is a modern React Single Page Application (SPA) built with Vite, utilizing Supabase for backend services (Auth, Database) and Google's Gemini AI for generative language features.

React Frontend (Vite)
↔
Supabase (PostgreSQL + Auth)
↔
Gemini AI API (@google/genai)
  • Frontend: React 18, Tailwind CSS, Framer Motion (animations), Lucide React (icons).
  • Backend/BaaS: Supabase handles user authentication, data persistence, and Row Level Security (RLS).
  • AI Services: Gemini 3.1 Flash is used for dynamic chat, lesson generation, and vocabulary corrections.

2. Directory Structure & Element Placement

Where everything sits in the repository:

📁 /src (Frontend Source)
  • App.tsx: The monolithic core of the application. Contains routing logic (tabs), main state, static lesson definitions, and the primary UI layouts.
  • main.tsx: React entry point.
  • index.css: Global Tailwind CSS imports and base styles.
  • types.ts: Global TypeScript interfaces (Lesson, UserProfile, ChatMessage, etc.).
📁 /src/components (UI Components)
  • Quiz.tsx: The interactive quiz component used at the end of lessons.
📁 /src/services (External Integrations)
  • geminiService.ts: Encapsulates all calls to the Google Gemini API. Handles chat sessions, prompt engineering, and response parsing.
📁 /src/lib (Utilities)
  • supabase.ts: Initializes and exports the Supabase client.
  • utils.ts: Contains the cn() utility for merging Tailwind classes.
📁 /supabase (Database & Backend)
  • /migrations/00001_initial_schema.sql: The complete SQL script to generate tables, RLS policies, and triggers.
📁 /docs (Documentation)
  • DATABASE_DESIGN.md: Schema documentation.
  • SUPABASE_RLS.md: Security policies documentation.
  • USER_MANUAL.md & TEST_MANUAL.md: Guides for users and QA.

3. Data Flows & Logical Flows

A. Authentication & Session Flow
User Enters Email/Pass
→
Supabase Auth API
→
Trigger: handle_new_user()
→
App.tsx onAuthStateChange

Logic:

  1. User signs up or logs in via the UI in App.tsx.
  2. Supabase handles the credential verification.
  3. If it's a new signup, a PostgreSQL trigger (on_auth_user_created) automatically creates a row in the profiles table.
  4. The onAuthStateChange listener in App.tsx detects the session, sets the user state, and fetches the user's profile and custom lessons.
  5. Password Recovery: Handled via OTP. User requests reset -> Supabase sends email -> User enters 6-digit code -> App calls verifyOtp -> User sets new password.
B. Lesson Data Flow (Static vs. Custom)

Static Lessons:

  • Location: Hardcoded in App.tsx as the INITIAL_LESSONS array.
  • Audio/Video: YouTube URLs are embedded using the VideoPlayer component. Audio pronunciations (if added) would rely on browser TTS or external audio files linked in the lesson object.
  • Flow: Loaded directly into the lessons state on boot.

Custom Lessons:

User Requests Topic
→
geminiService.generateLesson()
→
Supabase Insert (lessons table)
→
App UI Updates
  • Location: Stored in the Supabase lessons table with is_static = false.
  • Flow: Fetched via fetchCustomLessons() in App.tsx and appended to the static lessons list.
C. AI Chat & Tutor Flow
User Types Message
→
App.tsx handleSendMessage()
→
geminiService.chat()
→
Gemini API
→
Response Rendered (Markdown)

Logic:

  1. The user selects a Tutor (defined in the TUTORS array in App.tsx).
  2. The tutor's personality is injected into the Gemini System Prompt via geminiService.ts.
  3. Messages are sent to the Gemini API. The chat history is maintained in the chatHistory state to provide context to the AI.
  4. Responses are rendered using react-markdown to support formatting.
D. Settings & Offline-First Sync Flow
User Changes Setting (e.g., Speed)
→
React State Updates
→
localStorage.setItem()
→
Supabase Update (profiles)

Logic:

  • Configurations like playback_speed, is_sound_enabled, and selected_tutor_id are initialized from localStorage for instant loading.
  • A useEffect hook in App.tsx watches these states. When they change, it immediately updates localStorage and asynchronously updates the Supabase profiles table.
  • When the user logs in on a new device, the profile fetch overwrites the local state, syncing their preferences.
